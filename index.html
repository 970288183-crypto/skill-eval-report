线上对照版（对照组）
现有线上多轮流程
基于人工配置的多轮对话流程，每个意图路径均需运营逐节点配置。回复话术基于固定模板，覆盖场景内的标准路径。
📌 实验本质：基准线，衡量其他两组的改变幅度
📄
文档转译 skill（实验组1）
基于 LLM + 原始SOP文档
将原始SOP文档直接转译为 skill 知识库，大模型基于语义理解动态匹配意图、调用工具获取实时数据、生成回复。知识来源最完整，开发成本最低。
📌 转译源：SOP文档 → 一次转译 → skill 知识库
🔄
流程转译 skill 版（实验组2）
基于 LLM + 已有流程结构
将现有多轮流程转译为 skill 格式，保留原有流程层级和节点关系。大模型在已有流程内基于语义理解回复，不需重写话术。转译成本中等。
📌 转译源：SOP → 流程 → skill（两次转译）

实验结论：一句话看懂
踏过详细数据，直达核心判断
✅ 路径验证
大模型 skill-agent 架构具备取代多轮流程的运行能力
识别准确率显著高于对照组（+24pp），回复展现更友好（+12pp），对话效率持平，证明路径可行。
⚠️ 待破题
上线前需解决风险输出和完整性两个问题
实验组风险内容出现率 8–10%，方案信息不完整是目前最大痛点，两项均为可修复的工程问题。
🏆 初版建议首选方案
文档转译 skill 是当前综合天花板最高的方案
识别、友好度均显著优于对照组，且知识来源最完整、可维护性最高；解决力略低的原因已明确（少查询），修复后预期效果可超越对照组。
📌 尚需注意
完全解决率未超越对照组（符合预期）
初版已验证路径可行性，完全解决率的提升需要【查询工具调用】和【回复限制】均进一步调优。

维度一：识别能力对比
三组在「理解用户意图」上的差异与原因
识别结果构成分布（三组对比）
每组 50 条 session；绿=识别准确，黄=部分准确；三组均无识别错误
📊 数据结论
① 识别准确率：文档转译skill（86%） > 流程转译skill版（80%） > 线上对照版（62%）
② 三组均无「识别错误」，差异集中在「部分准确」占比——对照组高达 38%，两实验组分别为 14% 和 20%
🔍 原因分析
「部分准确」指主意图识别正确，但对意图变化或补充信息理解有偏差。固定多轮流程只能识别单场景主意图，用户追问/转变诉求时流程节点无法动态跟进，大量落入「部分准确」。LLM-based skill 基于语义理解，天然具备多轮动态识别能力
相较线上对照版的识别提升幅度
以对照组（62%）为基准，展示两实验组识别准确率提升 pp 值
🏆 三组对比结论
① 文档转译skill 提升最明显（+24pp）——SOP文档保留完整原始语义，识别容错性最高
② 流程转译skill版 提升也较显著（+18pp），但流程在「人工→多轮节点」转换时已有信息损耗，对非标意图的覆盖不及文档转译版
💡 建议优化方向
对照组存在结构性识别瓶颈；两实验组可进一步优化 skill 中对「意图澄清」和「关联场景补充」的识别引导，降低剩余「部分准确」占比
📌 识别能力天然优劣势：三条路径能识别到什么程度？
线上对照版（多轮流程）
✅ 优势
稳定性 · 预设场景内意图识别稳定可控，不会出现方向性偏差
❌ 劣势
能力边界 · 只能识别提前配置好的意图路径，较难灵活承接用户的个性化追问
可持续性 · 运营侧持续扩充意图规则的边际效益递减——规则越加越多维护成本线性增长，但遇到新表达或边缘意图仍会断路；这是规则匹配模式的结构性上限
文档转译 skill（实验组1）
✅ 优势
动态识别 · 基于语义理解动态感知意图，不依赖固定路径——用户追加信息、中途转变诉求、换个说法，都能灵活承接
知识完整性 · 知识来自原始SOP，覆盖面最完整，识别容错性最高
❌ 劣势
意图边界模糊 · 语义越灵活，意图边界越模糊——对相似但不同的意图，模型有时可能判断不准
流程转译 skill 版（实验组2）
✅ 优势
动态识别 · 语义驱动，识别能力优于固定流程
场景边界清晰 · 保留了流程的场景边界，减少泛化误判风险
❌ 劣势
知识损耗 · 流程转译带来的知识损耗，导致非标意图覆盖范围窄于文档转译版
质量上限 · 识别上限受制于原始流程的设计质量

维度二：解决力对比
三组在「给出有效方案」上的差异与根因
解决力主标项分布（三组对比）
完全解决 / 部分解决 / 未解决；每组 50 条 session
📊 数据结论
完全解决率：线上对照版（52%） > 文档转译skill（48%） > 流程转译skill版（46%），差距在 6pp 内，说明方案完整性是三组共同短板
关键差异：流程转译skill版出现2例未解决（4%），文档转译skill未解决为0，说明流程转译版在极端case兜底能力最弱
🔍 各版本部分解决/未解决的核心原因
线上对照版
· 主因：识别偏差引发方案方向错误——上游意图识别不准导致回复答非所问
· 次因：流程配置缺陷或模型幻觉导致方案不准确（已与运营确认）
文档转译 skill
· 主因：缺少代查询信息——未主动查询粉丝量、实名认证结果、账号违规情况等关键状态
· 次因①：LLM幻觉——错误引导用户提供用户ID、误改反馈时效、编造方案内容
· 次因②：查询返回字段缺乏语义说明——skill未解释API返回字段含义和取值范围，不同模型对同一字段理解不一致，即使查到数据也可能因字段误读输出错误内容（两实验组共性）
流程转译 skill 版
· 主因：缺少代查询信息（与文档转译版相同，两实验组共性短板）
· 次因①：存在缺少关键链接、错误引导用户提供用户ID等问题
· 次因②：查询返回字段缺乏语义说明（同上，两实验组共性问题）
问题标签分布（三组对比）
部分解决/未解决的具体问题分类，可多选，按绝对数量统计
🏆 三组对比结论
① 方案信息不完整是三组共同短板
· 方案信息不完整是三组最高频问题（23–26条），不因版本而异，属系统性能力缺口
· 两实验组集中在少查询，根源是 skill 触发代查工具时机不准确（大部分为应查未查）
· 对照版的「部分解决」更多来自识别层误判传导，属上游问题下沉
② LLM 回复相关性显著优于固定流程
· 两实验组均无答非所问，对照版出现 1次
· 说明 LLM-based skill 在回复相关性上具有结构性优势
③ 流程转译版存在极端case兜底失效
· 方案错误（1次）：输出错误内容，归因为流程翻译的知识转换损耗
· 无法提供方案（1次）：归因尚存疑，可能是服务接口调用失败或skill知识覆盖盲区，需结合session日志进一步排查
💡 建议优化方向
① P0：补充代查询工具触发规则，识别到粉丝量/实名认证/账号违规等查询需求时自动触发
② 字段语义标注：在skill知识配置中补充API返回字段的含义说明和枚举值解释
③ 幻觉管控：在prompt中增加事实约束指令，禁止模型生成无法核实的信息

📌 方案回复与解决力天然优劣势：谁能给出更好的答案？
线上对照版（多轮流程）
✅ 优势
可预期性 · 方案经人工配置→审核→测试，输出完全可预期，几乎无幻觉，无风险内容
覆盖准确 · 标准场景内的解答完整且准确
❌ 劣势
用户侧 · 只能回答预先配置好的问题——用户问偏、追问或换种说法，即无方案可给
运营侧 · 业务规则变化后，每个流程节点都要手动更新，运营维护成本极高
文档转译 skill（实验组1）
✅ 优势
灵活承接 · 基于语义理解，能承接个性化追问——用户追加信息、转变诉求、换种说法均可跟进
知识完整性 · 直接来自原始SOP，覆盖面最完整，可应对更多场景变体
可维护性 · 业务更新只需重跑一次转译，维护成本远低于逐节点手动配置
❌ 劣势
幻觉风险 · 工具调用策略不完善时，可能编造方案或引用错误信息
流程转译 skill 版（实验组2）
✅ 优势
灵活性 · 语义驱动，灵活性优于固定多轮流程
对话效率 · 保留流程精简结构，对话路径短，效率高
❌ 劣势
知识损耗 · 经历 SOP→流程→skill 两次转换，信息损耗最大，方案深度最弱
质量上限 · 原始流程的缺陷会被原样继承，能力上限受制于流程本身的设计质量
幻觉风险 · 幻觉风险同样存在，且知识量更少，模型更难以正确知识抑制幻觉

交互友好度与对话效率
交互友好度对比
「好」占比越高越优；每组 50 条 session
▮ 文档转译 skill 94%
▮ 流程转译 skill 版 86%
▮ 线上对照版 82%
现状：三组交互友好率排序为文档转译skill（94%）> 流程转译skill版（86%）> 线上对照版（82%），文档转译版比对照组高 12pp，差距明显。

原因：
1. 对照版依赖固定话术模板，回复灵活度低，用户感知差；
2. LLM-based skill 基于语义理解，能根据用户当前情绪和措辞动态调整表达，在情绪安抚、措辞清晰度、追问承接上显著优于规则流程。

需说明的是：友好率衡量的是回复的表达质量，与「是否解决了问题」是两个不同维度——文档转译版友好率 94% 但完全解决率仅 48%，说明大模型在方案表达和呈现上更友好清晰，但表达优秀的前提是工具调用和知识覆盖到位，两者需协同提升。
对话效率对比
三组差异细微（仅 2pp）；横坐标从 88% 起步，使细微差距更易读
现状：三组效率几乎一致（92–94%），差距仅 2pp，无显著差异。

需说明的是：
1. 对照版（多轮流程）因识别偏差或回复不完整，实际交互中存在用户反复追问、澄清的情况，交互费力度理论上更高；
2. LLM-based skill 基于语义理解，本应在减少无效追问和澄清问题上更有优势。
三组当前数值接近，说明第一版 skill 在对话效率上尚未拉开差距（在预期之内），初版主要验证路径可行性，效率优化需在工具调用和知识覆盖完善后进一步体现。

后续关注：随着「少查询」问题修复、方案完整性自检优化，预期用户追问次数应有明显下降，对话效率指标预期在下一版本中显著优于对照组。

产研优化优先级建议
按优先级排列，P0 为上线前必须解决
优先级问题影响范围具体表现优化方向
P0 立即处理 风险内容输出 文档转译 8%、流程转译 10% LLM 幻觉导致不当内容（内网链接/过度承诺等） 上线前必须增加 output guardrail 过滤层
P0 立即处理 方案信息不完整 三组均超 46%（部分解决率） 用户需反复追问才能获取完整信息 Prompt 增加「完整性自检」，要求模型检查关键字段是否覆盖
P1 迭代优化 少查询 三组均有（6–9次） 缺少代查询信息，用户需自行操作 Skill 中补充工具调用触发规则，识别到"查询类"时自动触发
P1 迭代优化 查询字段语义缺失 两实验组共性问题 API返回字段无语义说明，模型解读不一致，查了也可能输出错误内容 在 skill 知识配置中补充每个返回字段的含义和枚举值解释
P1 迭代优化 流程转译知识盲区 流程转译版：方案错误1次、无法提供方案1次 流程翻译知识丢失或接口异常 增加知识一致性校验，排查接口稳定性，补充 fallback 兜底策略
P2 持续改善 完全解决率整体偏低 三组完全解决率仅 46–52% 超过一半 session 仅部分解决 扩展知识库覆盖范围，提升工具调用成功率，优化多轮追问策略
